AI 코딩 에이전트 × 기존 테스트 자산

AI에게 TDD를 시키는 것만으로 충분할까?

  • AI가 코드를 빨리 만들수록, 무엇이 맞는지 판단하는 기준이 더 중요해진다.
  • 테스트는 오류 검사를 넘어 업무 의도와 완료 조건을 AI에게 전달한다.
  • Frontend와 Backend는 테스트 파일이 아니라 API 계약과 핵심 시나리오를 공유한다.
  • 작은 계약 API와 Repository Harness로 이 기준이 실제로 작동하는지 확인한다.

Kent Beck · AI가 잘못된 방향으로 가는 3가지 신호

01 · 같은 행동을 반복한다 같은 시도와 실패를 반복하며 진전이 없으면 개입한다.
02 · 요청하지 않은 범위를 구현한다 논리적인 다음 단계여도 승인하지 않은 기능은 범위 밖이다.
03 · 테스트를 삭제하거나 우회한다 삭제·비활성화·기대값 변경은 즉시 확인한다.

Start Here · 먼저 볼 기준

테스트가 있다는 것과 AI가 올바르게 검증하는 것은 다르다

  • 현재 테스트 구조의 장점과 운영 비용을 함께 본다.
  • 테스트 수보다 변경에 맞는 반복 검증에 집중한다.
오늘의 목표

AI가 변경한 코드가 기대한 업무 규칙을 지키는지 재현 가능한 테스트와 실제 HTTP 응답으로 증명한다.

Strengths (장점)

  • Fast feedback잘못된 입력을 DB 작업 전에 발견한다.
  • Layered safetyunit · integration · e2e · harness가 서로 다른 위험을 보호한다.
  • Executable spec업무 규칙이 반복 실행 가능한 명세가 된다.

Trade-offs (고려사항)

  • Execution costDB·인증·외부 연동이 있는 테스트는 실행 비용이 크다.
  • Impact discovery변경 경로와 가까운 테스트를 먼저 찾는다.
  • Mock과 실제 경계실제 연동 차이는 integration·e2e로 보완한다.

01 / EXPECTED

기대 결과 정하기

  • 사람이 동작과 기대값을 정한다.
  • plan.md에서 테스트 하나를 고른다.
02 / RED

실패 확인하기

  • AI가 테스트 하나를 작성한다.
  • verify 명령은 focused 실패에서 중단한다.
  • 사람이 동작 부재로 실패했는지 확인한다.
03 / GREEN

최소 코드 수정하기

  • AGENTS.md 규칙 3~5에 따라 실패한 테스트는 유지한다.
  • 최소 제품 코드 수정 후 같은 verify 명령을 다시 실행한다.
04 / VERIFY

전체 확인하기

  • focused가 통과하면 full·lint·type을 이어서 실행한다.
  • .harness/latest-run.json에서 실행 결과를 확인한다.

CASE 01 · FIELD BOUNDARY (필드 경계)

Payment Terms (지급조건 검증)

  • 지급 비율 합계는 100이어야 한다.
  • 개별 비율이 음수면 422로 거절한다.
def test_rejects_negative_payment_percentage_even_when_sum_is_100():
    response = client.post(
        "/contracts/payment-terms/validate",
        json={"advance_payment_percentage": -10, ...},
    )
    assert response.status_code == 422

03 · Why TDD Here

우리 회사에 TDD가 필요한 이유와 적용 범위

  • 운영 장애를 모두 테스트로 없앨 수는 없다.
  • 코드로 재현 가능한 오류는 배포 전에 반복 검증할 수 있다.
WHY TDD · 필요한 이유

현재 구조에서 테스트가 효과적인 세 지점

멀티레포 경계·데이터 제약·리팩토링 회귀는 코드와 테스트로 반복 확인할 수 있다.

01 · 멀티레포 구조

저장소마다 기대 동작을 고정해야 한다

각 저장소의 기대 동작과 연결 계약을 테스트로 고정한다.

02 · 숨은 데이터 제약

코드만 보고 알기 어려운 조건이 있다

기존 DB 데이터·내부 validator·상태 조합을 테스트 입력으로 재현한다.

03 · 리팩토링·디버깅

오류 재현이 수정의 기준이 된다

실패 테스트로 오류를 재현해 원인과 재발을 함께 확인한다.

LIMITS · 적용 한계

TDD를 과하게 기대하면 안 되는 세 가지

TDD는 한 검증 계층이다. 운영·통합·배포 문제까지 혼자 해결하지 않는다.

01 · 멀티레포 전체 해결

저장소 연결 자체를 해결하지는 않는다

배포 순서·버전·권한 연결은 계약 관리와 CI가 함께 다룬다.

02 · 운영 환경 완전 복제

실제 DB와 외부 환경을 모두 재현하지 못한다

운영 DB·네트워크·외부 서비스는 integration·E2E·모니터링으로 보완한다.

03 · 모든 코드 Test-first

위험이 낮은 코드까지 강제하지 않는다

단순 DTO·framework 동작보다 제약·규칙·오류·리팩토링부터 적용한다.

Closing

AI 시대의 TDD는 테스트 대필 기술이 아니다. 사람의 의도를 실행 환경으로 만든다.
