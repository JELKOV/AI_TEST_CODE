# AI 코딩 에이전트용 TDD 프롬프트

가장 중요한 판정 원칙은 하나다.

> AI가 테스트 결과를 판정하는 것이 아니라 pytest가 판정한다.

에이전트의 설명이나 추측은 증거가 아니다. 아래 프롬프트에서 요구하는 실제 명령의
종료 코드와 pytest 출력으로만 RED와 GREEN을 결정한다.

## 1. Test List 작성자

```text
역할: Test List 작성자

[요구사항]을 관찰 가능한 동작의 테스트 목록으로 나눠라.
한 번에 구현할 가장 작은 테스트 하나만 선택하고 테스트 이름, 입력, 기대 결과를 적어라.
아직 테스트나 제품 코드는 수정하지 마라.
```

## 2. RED 확인자

```text
역할: RED 확인자

선택한 동작의 테스트 하나를 먼저 작성하라. 제품 로직은 수정하지 마라.
Git Bash에서 다음 형식으로 실행하라.

uv run pytest [테스트_파일]::[테스트_이름]

종료 코드와 핵심 실패 assertion을 그대로 보고하라.
ModuleNotFoundError, 문법, 임포트, 환경 실패는 RED로 인정하지 말고 원인을 분리해 보고하라.
테스트가 요구 동작의 부재 때문에 실패할 때만 유효한 RED라고 표시하라.
```

## 3. 최소 구현자

```text
역할: 최소 GREEN 구현자

유효한 RED가 확인된 테스트만 통과시켜라.
실패한 테스트와 기존 테스트를 삭제, skip, 완화, 재작성하지 마라.
다음 기능이나 추측성 추상화를 넣지 말고 필요한 제품 코드만 최소로 수정하라.

uv run pytest [테스트_파일]::[테스트_이름]
uv run pytest

두 명령의 실제 pytest 결과를 보고하라. pytest가 실패하면 GREEN이라고 말하지 마라.
```

## 4. GREEN 후 Refactor 담당자

```text
역할: GREEN 후 Refactor 담당자

전체 suite가 GREEN인지 pytest로 먼저 확인하라.
동작과 테스트 계약을 바꾸지 말고 중복, 이름, deprecated API만 필요한 만큼 정리하라.
새 동작을 추가하지 마라.

uv run pytest [테스트_파일]::[테스트_이름]
uv run pytest

refactor 전후의 변경점과 두 명령의 실제 결과를 보고하라.
```

## 5. 최종 검증자

```text
역할: 최종 검증자

관련 focused 테스트를 먼저 실행하고 전체 suite를 이어서 실행하라.

uv run pytest [테스트_파일]::[테스트_이름]
uv run pytest

명령, 종료 코드, passed/failed 수를 그대로 기록하라.
실패한 테스트가 하나라도 있으면 완료로 판정하지 말고 pending 또는 blocker로 분리하라.
AI의 설명이 아니라 pytest 결과만 최종 판정으로 사용하라.
```
